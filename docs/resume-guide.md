# 简历与面试表达指南

## 项目名称

**校园活动管理与动态签到平台｜Django 全栈项目**

## 简历项目描述（可直接使用）

技术栈：Python、Django、SQLite/PostgreSQL、Bootstrap、Django Template、GitHub Actions

- 独立设计并实现校园活动管理平台，打通活动发布、在线报名、限时签到、数据统计与 CSV 导出的完整业务闭环。
- 基于 Django Auth 与 Profile 角色模型实现学生、活动老师、开发者三级 RBAC，并结合资源所有权校验控制活动编辑和签到名单访问。
- 使用数据库事务、行级锁与联合唯一约束处理满额及重复报名；生产环境切换 PostgreSQL 后可避免并发请求造成名额超卖。
- 使用 `secrets` 生成动态签到码并配置签到时间窗，将签到/撤销动作以 JSON 审计日志记录 IP、User-Agent 和时间戳，支持操作追溯。
- 编写 16 个自动化测试覆盖权限、报名、签到、审计和导出等核心场景，并接入 GitHub Actions 持续集成。

> 不要写“服务数千用户”“性能提升 80%”等没有压测或线上数据支撑的数字。可量化内容优先使用真实的角色数、业务环节数、测试数和覆盖场景。

## 30 秒项目介绍

这是一个面向校园活动场景的 Django 全栈系统。我没有只做 CRUD，而是围绕真实业务补齐了三级角色权限、活动所有权校验、满额报名并发控制、限时动态签到和可追溯审计日志。项目目前有 16 个自动化测试并接入 CI，本地使用 SQLite，部署时可以通过环境变量切换 PostgreSQL。这个项目让我完整实践了从数据建模、权限边界到测试和工程化交付的过程。

## 高频追问

### 为什么报名要加事务和行级锁？

如果两个请求同时读到“还剩 1 个名额”，普通的先查询再创建会让两者都成功。事务配合 `select_for_update()` 会在 PostgreSQL 中锁住该活动记录，使同一活动的报名请求依次执行；联合唯一约束则防止同一用户重复报名。

### 为什么审计日志用 JSONField？

签到日志字段结构明确但可能扩展。JSONField 能保存动作、时间、IP 和终端信息并保持结构化，比拼接文本更容易验证和后续迁移。若日志量增大，可再拆成独立 AuditLog 表以支持索引和查询。

### 这个项目目前最大的不足是什么？

当前是单体 Django 应用，SQLite 仅适合本地开发；尚未加入缓存、异步任务和真实线上压测。下一步会优先部署 PostgreSQL，并用 Locust 验证热点活动并发报名，再根据结果决定是否加入 Redis，而不是为了技术栈数量盲目拆分。

### 为什么不用 JWT 或前后端分离？

该项目以服务端渲染为主，Django Session 与 CSRF 已能清晰、安全地覆盖浏览器端场景。若后续需要小程序或移动端，再抽出 REST API 并引入 Token 认证，当前选择可以控制复杂度。
